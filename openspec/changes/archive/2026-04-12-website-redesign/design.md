## Context

The current capoeira-oldenburg.de is a single-scroll page hosted on Jimdo. It loads slowly, is hard to navigate on mobile, and buries the training schedule -- the single most important piece of information for converting visitors into trial attendees. The site will be replaced entirely with a static site generated by Hugo and deployed via GitHub Pages on the same custom domain.

There is no existing codebase. This is a greenfield project.

## Goals / Non-Goals

**Goals:**
- Fast, mobile-first, multi-page static site with nine pages
- Training schedule (Trainingszeiten) as the primary conversion page -- immediately findable and scannable
- Markdown-based content authoring for blog posts and events
- Proper SEO for local search ("Capoeira Oldenburg")
- Automated build and deploy via GitHub Actions to GitHub Pages
- All existing content migrated from the current site

**Non-Goals:**
- Multilingual support (German only for now)
- Dedicated photo/video gallery
- RSS feed
- CMS or web-based editing interface
- Contact form
- Client-side JavaScript interactivity beyond what Hugo provides

## Decisions

### D1: Static Site Generator -- Hugo

**Choice:** Hugo

**Alternatives considered:**
- Astro: Modern component model, but requires Node.js and adds toolchain complexity unnecessary for a pure content site.
- 11ty: Flexible but less structured; requires more manual setup for content sections that Hugo provides natively.
- Jekyll: Ruby dependency, slower builds, less active development.

**Rationale:** Hugo's content model (sections, front matter, data files, taxonomies) maps directly to the nine-page site with blog and events. Single binary, no runtime dependencies. The Go template syntax is the main trade-off but manageable given the low template complexity.

### D2: CSS Approach -- Plain CSS with Custom Properties

**Choice:** Hand-written CSS using custom properties for design tokens, processed through Hugo Pipes for minification and fingerprinting.

**Alternatives considered:**
- Tailwind CSS: Productive but adds a Node.js build dependency, contradicting the zero-dependency benefit of Hugo.
- PicoCSS / Simple.css: Quick defaults but limits the distinctive visual identity appropriate for a Capoeira Verein.

**Rationale:** The site has nine pages with straightforward layouts. Custom properties provide consistent design tokens (colors, spacing, typography). Mobile-first via `min-width` media queries. No build dependencies beyond Hugo itself.

### D3: Events as Markdown Files

**Choice:** Each event is a markdown file in `content/events/` with front matter (date, title, location). The list template splits upcoming vs. past events using Hugo's `now` function compared to the event date.

**Alternative considered:**
- Single YAML data file: Compact but does not support rich event descriptions or leverage Hugo's content model.

**Rationale:** Consistent authoring experience with blog posts. Supports event descriptions, images, or recap text. Hugo handles the content natively.

### D4: Training Schedule as YAML Data File

**Choice:** `data/trainingszeiten.yaml` containing structured entries (weekday, time, location, notes). The Trainingszeiten page template iterates over this data.

**Rationale:** Training times are structured data (not prose). A YAML file is easier to read and edit than embedding the schedule in markdown. The template renders it into a clear, scannable layout.

### D5: LocalBusiness JSON-LD in Base Template

**Choice:** A single `<script type="application/ld+json">` block in the base layout partial, present on every page.

**Alternative considered:**
- Selective inclusion via partial: Marginally cleaner but adds complexity for no real benefit.

**Rationale:** One location to maintain. Presence on legal pages is harmless. Simplest approach.

## Risks / Trade-offs

- **Content migration completeness** -- Existing content must be manually extracted from capoeira-oldenburg.de and converted to markdown/YAML. Risk: missing or malformatted content. Mitigation: dedicated migration task phase with manual review against the live site.
- **Legal text accuracy** -- Impressum and Datenschutzerklaerung must contain legally correct text per DDG SS5 and DSGVO Art. 13/14. Mitigation: content is a task concern; the architecture ensures the pages exist and are linked from every page via the footer. Actual legal text must be reviewed by the Verein.
- **DNS cutover** -- Switching the custom domain from Jimdo to GitHub Pages requires DNS changes. Risk: brief downtime during propagation. Mitigation: lower TTL in advance; verify GitHub Pages configuration before DNS switch.
- **Hugo Go template learning curve** -- Minor. The templates needed are straightforward (range, if, with, partial). No deep Hugo expertise required.
- **No dynamic event filtering** -- The upcoming/past event split is determined at build time. If an event passes its date between builds, it will remain in "upcoming" until the next deploy. Mitigation: acceptable given the low update frequency. A scheduled GitHub Actions rebuild (e.g., weekly) can keep it current.
